2

IST Timezone Setup

The pipeline uses your Mac's system timezone for the APScheduler — the scheduled job triggers must fire at the correct IST times. If your Mac is set to a different timezone, the pipeline will pull at the wrong times.

Setting System Timezone to IST (Permanent)

zsh · System Preferences # Option A: macOS System Preferences (recommended) # → System Settings → General → Date & Time → Time Zone # → Set to "Asia/Kolkata" or "Mumbai" # Option B: Terminal command (requires admin) sudo systemsetup -settimezone "Asia/Kolkata" # Verify: date # Should show IST after the time

Alternative: Environment Variable Per Session

If you cannot change the system timezone (e.g., shared Mac), set the TZ variable in each terminal session before running the pipeline:

export TZ=Asia/Kolkata # Add this to the top of your ~/.zshrc for it to persist across all new sessions
ℹ️The API and dashboard are not time-sensitive in the same way. Only the pipeline's APScheduler needs IST alignment. The record_time values in BigQuery are already stored as IST — no conversion happens in the data path.

5

The Backfill Process

When you run run_pipeline.py for the first time (or after a fresh project setup), the backfill module runs automatically before the live scheduler starts. It pulls exactly 4 days of historical data: 2026-02-17 through 2026-02-20.

Why 4 Days?

Most analytical calculations — especially IVP (IV Percentile) which requires 90 days of IV data — will improve significantly with more history. However, the system is designed to be useful from day one with just 4 days of bootstrap data. As the live scheduler accumulates more trading days, IVP calculations become more accurate. IVP requires at least 10–15 trading days before the percentile rankings become reliable. The 4-day backfill is the minimum viable starting point, not the ideal steady state.

How the Backfill Works

  1. Check for Existing Files (Idempotent)
    For each trade date, checks if data/raw/options/trade_date=YYYY-MM-DD/NIFTY.parquet already exists. If yes, skips that date entirely. This means you can safely re-run run_pipeline.py without creating duplicate data or pulling redundant BigQuery rows.
  2. Pull One Full Day from BigQuery
    Uses WHERE DATE(record_time) = 'YYYY-MM-DD' to pull all rows for that calendar date. Each trading day typically contains 75 snapshots × ~2,000 rows = ~150,000 rows. The pull is a single large query; BigQuery processes it in 5–15 seconds depending on slot availability.
  3. Process and Write Parquet
    Runs the full validation + processing pipeline for the pulled data. Writes final Parquet files partitioned by underlying. For a full trading day, this produces 5 option Parquet files (one per index underlying) and 1 futures Parquet file per trade_date directory.
  4. Advance Watermark After Each Day
    After successfully writing a day's Parquet files, the watermark advances to the last record_time of that day (~15:30). This means if the backfill is interrupted after 2 days, re-running picks up from day 3, not from the beginning.
  5. Live Scheduler Takes Over
    After all 4 backfill days are processed, the watermark sits at approximately 2026-02-20 15:30:00. The APScheduler then starts its 5-minute live cycle. The first live pull (at 09:16:30 on the next trading day) picks up from that exact watermark. No gap, no overlap.

Expected Backfill Duration

Phase Duration Notes
BigQuery pull per day 5–20 seconds Depends on BigQuery slot availability and network
Processing + writing per day 10–30 seconds 4 days total ≈ 2–5 minutes of Parquet writing
Total backfill 2–5 minutes Expect it to be done before market opens
✅First-run tip: Start the pipeline the evening before your first trading day (e.g., the night before a Monday), so the backfill completes overnight and the morning live pull fires cleanly at 09:16:30.

Daily Routine

Once the system is running, your everyday workflow with the dashboard follows this structure. The system handles data collection automatically — your only responsibility is to keep the three processes running and open the browser before market opens.

PRE-MARKET

08:45 — 09:00 IST · System Check

08:45
Open three terminal tabs. Navigate to project root in each. Activate venv in Tab 1 and Tab 2.
08:47
Run python run_pipeline.py in Tab 1. Wait for "Live scheduler started. First pull at 09:16:30".
08:50
Run python run_api.py in Tab 2. Confirm Uvicorn shows "Application startup complete".
08:52
Run npm run dev in Tab 3 from the dashboard/ directory.
08:54
Open http://localhost:5173 in Chrome. The dashboard loads with yesterday's close data — this is expected and correct.
08:55
Set the date selector in the dashboard to today's date. Select the underlying you plan to trade (NIFTY, BANKNIFTY, or FINNIFTY).
MARKET OPEN

09:15 — 09:20 IST · First Data Arrives

09:15
Pre-open session. Dashboard still shows previous day's data. Environment Gate shows "Loading…" or yesterday's score.
09:16:30
Pipeline fires the first live pull. Fetches all rows since the watermark. ATM window is computed from the 09:15 snapshot spot price.
09:17
API picks up new Parquet files. Dashboard panels begin refreshing (TanStack Query detects new data within 5 seconds of the Parquet write).
09:20
All 8 panels should now show today's live data. Environment Gate should display a score. GEX chart shows the first data point. This is your entry window for Phase 2 analysis.
TRADING HOURS

09:20 — 15:30 IST · Active Monitoring

Every 5 min
Pipeline auto-pulls. All panels update within 10 seconds. You do not need to do anything manually.
09:30–10:30
Primary entry window. Check Gate score, V_CoC direction, and Strike Screener before entering positions.
If alert fires
Alert Feed (Panel F) shows a new entry. Review it immediately. Cross-check against your open positions in Panel H.
14:30–15:00
Expiry tier switches to NEAR (DTE ≤ 1 for weekly expiries). Screener will deprioritise these. No new entries after 14:45 (see Part 7).
CLOSE

15:30 — 15:45 IST · End of Day

15:31:30
Final pipeline pull of the day. Watermark advances to the 15:30 snapshot. Scheduler confirms no more pulls until next trading day at 09:16:30.
After 15:35
Review Position Monitor for final P&L attribution. Log any observations in your trading journal. The three processes can be left running or shut down (see below).
7

Shutting Down Cleanly

The system can be stopped at any time without data loss. All three processes are stateless between pulls — no in-memory state is lost when they're killed, because all state is on disk in Parquet files and the watermark JSON.

Clean Shutdown Sequence

# In Tab 3 (Dashboard) — press: Ctrl+C # In Tab 2 (API) — press: Ctrl+C # In Tab 1 (Pipeline) — press: Ctrl+C # APScheduler stops cleanly. If it's mid-pull, it completes the current batch first. # The watermark is already updated — no data is lost.

Overnight / Weekend Shutdown

It is safe to leave the Mac sleeping with all three processes stopped between trading sessions. On the next trading day, start all three tabs again before 09:15 IST. The backfill module will not re-run (files already exist for previous days), and the live scheduler fires cleanly from the last recorded watermark.

✅Recommended: Add a simple shell alias in ~/.zshrc to start all three processes with one command. Or use iTerm2 profiles to auto-launch all three tabs with the correct commands.
