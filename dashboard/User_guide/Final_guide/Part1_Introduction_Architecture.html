What This Dashboard Does

The Option Buying Dashboard v2.0 is a personal, locally-hosted quantitative analytics system that gives a retail option buyer access to the same market structure signals that institutional desks use — without any subscription fees, without cloud processing delays, and without counterparty risk from third-party signal services.

Every five minutes during NSE market hours, the system automatically: queries your BigQuery data source for fresh option and futures snapshots, computes over a dozen derived analytical metrics per row, ranks all eligible strikes by a slippage-adjusted efficiency score, and updates eight dashboard panels in your browser without any manual intervention.

The Core Problem This Solves

Retail option buyers most commonly fail for three structural reasons:

❌
Buying in wrong regimes

Buying options when market makers hold large positive gamma positions causes the market to mean-revert against every directional move. The dashboard's GEX panel makes this regime visible in real-time so you sit out pinning days.

❌
Ignoring institutional flow

Retail traders act on price — but institutions build futures positions 60–90 minutes before price moves. The CoC Velocity panel reads this institutional footprint from futures carry data, giving you an early warning system.

❌
Poor strike selection

Choosing a strike based on "looks cheap" ignores slippage, theta burden, and order book quality. The S_score formula in the Strike Screener ranks every eligible strike on all three factors simultaneously.

System Architecture

The system runs as three fully independent processes simultaneously on your Mac. Each process has exactly one responsibility. They communicate only through files on disk — there are no internal APIs between them, no shared state, no message queues. This makes the system transparent, robust, and easy to debug: any layer can be stopped and restarted without affecting the others.

🔄
Terminal 1 — Pipeline

Pulls raw data from BigQuery every 5 minutes. Validates, processes, and writes Parquet files to disk. Manages the watermark for incremental pulls. The only process that ever contacts BigQuery. All other processes are entirely local.

⚡
Terminal 2 — API

FastAPI reads Parquet files through DuckDB views. Executes all analytical SQL (GEX sums, CoC velocity, S_score ranking, IVP percentiles) and exposes the results as JSON at localhost:8000. Never writes anything.

🖥
Terminal 3 — Dashboard

Vite + React polls the API every 5 seconds using TanStack Query. Renders all eight panels at localhost:5173. Pure display layer. No computation, no data storage, no connections to anything except the local API.

ℹ️BigQuery is read-only. The service account credentials file (universal-ion-437606-b7-5629bd4bc421.json) is used only to run SELECT queries. Nothing is ever written back to BigQuery. Your source data cannot be modified by this system under any circumstances.

Why Parquet + DuckDB Instead of a Live Database?

Several more conventional architectures were considered and rejected:

ApproachProblemOur Solution
Query BigQuery directly from API300–1500ms latency per query. Dashboard would be sluggish. Also burns BigQuery slot quota. API becomes dependent on network.Pipeline pre-caches data to Parquet on local disk. API queries locally — typically 5–20ms per endpoint.
PostgreSQL or SQLite intermediate DBRequires ETL layer to load Parquet into DB. Schema migration complexity. Extra process to manage.DuckDB reads Parquet directly via glob views — zero ETL. New files are instantly visible without re-loading.
In-memory state in the APIAPI restart loses all data. No persistence between sessions. Complex invalidation logic.Everything is on disk in Parquet. API is completely stateless — restart it anytime without losing a single data point.

Data Flow — The 7-Step Cycle

Every five minutes during NSE market hours — first at 09:16:30 IST, then every 5 minutes until the final pull at or before 15:31:30 IST — the pipeline executes this exact sequence:

  1. Load Watermark
    Reads data/watermark.json to find the timestamp of the last successfully processed record. After the initial backfill, this starts at approximately 2026-02-20 15:30:xx. This timestamp is the exclusive lower boundary — all BigQuery rows with a record_time strictly after this value will be pulled. On the very first run, an initial watermark of 2026-02-16 15:30:00 is used, which causes the backfill to pull from 2026-02-17 onward.
  2. Pull Incremental Rows from BigQuery
    Executes: SELECT [all columns] FROM `universal-ion-437606-b7.bgquery.upxtx` WHERE record_time > TIMESTAMP("{watermark}") ORDER BY record_time ASC. Uses the service account JSON for authentication. A typical 5-minute snapshot returns 1,600–2,400 rows covering all NSE index option contracts (CE + PE for NIFTY, BANKNIFTY, FINNIFTY, MIDCPNIFTY, NIFTYNXT50) plus all index and stock futures. The pull has exponential-backoff retries (up to 4 attempts) via tenacity to handle transient BigQuery API errors.
  3. Validate & Clean
    The validator checks that all required schema columns exist in the result. It then coalesces null values in the ltp column: effective_ltp = ltp → close_price → close (first non-null wins). Negative IV values are corrected to 0. Whitespace is stripped from string columns. Row counts are logged by instrument type (OPTIDX, FUTIDX, FUTSTK). If fewer than 10 rows are returned during market hours, a warning is logged but processing continues.
  4. Load or Compute the ATM Window
    Checks whether data/processed/atm_windows/YYYY-MM-DD.json already exists for today. If yes, loads it (the window is fixed for the whole day). If this is the first pull of a new trading day, it computes the ATM window from the 09:15 snapshot: rounds the day-opening spot price to the nearest strike interval, then sets a window of ±8 strikes around that ATM. This produces 17 strikes per underlying and is saved immediately so all subsequent pulls reuse the same window.
  5. Compute All Derived Columns
    The processor adds these columns to each row:
    • snap_time — floored to 5-minute boundary (e.g. 09:20:00)
    • trade_date — date portion of record_time
    • expiry_tier — TIER1_NEAR (DTE ≤12), TIER1_FAR (DTE ≥35), or TIER2_IV_ONLY
    • in_atm_window — boolean, true if this strike is within today's ±8 window
    • moneyness_pct — (strike − spot) / spot × 100
    • gex_k — per-row GEX contribution: Gamma × OI × Spot² × 0.01 × direction_factor
    • coc — Cost of Carry: futures_ltp − spot (for futures rows)
    • oi_delta — change in OI from previous snapshot
    • obi_raw — (total_buy_qty − total_sell_qty) / (total_buy_qty + total_sell_qty)
    • dte — (expiry_date − trade_date).days
  6. Write Parquet Files to Disk
    Options data is written as data/raw/options/trade_date=YYYY-MM-DD/UNDERLYING_HHMM.parquet — one file per underlying per snapshot. For example, at 10:15 on 2026-02-24: NIFTY_1015.parquet, BANKNIFTY_1015.parquet, etc. Futures are combined into data/raw/futures/trade_date=YYYY-MM-DD/ALL_FUTURES_HHMM.parquet. All files use Snappy compression and row_group_size=100,000 — a critical performance setting that enables DuckDB to parallelize reads 3–5× faster than the default 10,000.
  7. Refresh DuckDB Views & Advance Watermark
    DuckDB's glob-pattern views (read_parquet('data/raw/options/**/*.parquet', union_by_name=true)) automatically discover the newly written files — no explicit refresh command is needed. The watermark is then advanced to the maximum record_time in the just-processed batch and written atomically to data/watermark.json via a .tmp file rename (guaranteeing no partial state even if the process crashes mid-write on macOS APFS).
ℹ️Time handling: The record_time column in BigQuery reads as "2026-02-20 10:05:19 UTC" but the values are already IST — no timezone conversion is performed. The pipeline strips the UTC label without shifting: pd.to_datetime(df["record_time"]).dt.tz_localize(None). All comparisons with market hours (09:15, 15:30) are made directly against these naive IST-valued timestamps.

Design Principles

These principles governed every architectural and code decision. Understanding them helps you extend or debug the system correctly.

P1

Idempotency — Safe to Re-run Anything

Every operation can be repeated without creating duplicates or corruption. The backfill checks for existing Parquet files before pulling. The watermark prevents duplicate BigQuery rows from ever being processed. DuckDB views are created with CREATE OR REPLACE. If the pipeline crashes at any point, re-running it from scratch is always safe.

P2

No Hardcoded Values Outside config.py

Every threshold, path, schedule time, underlying name, strike interval, and row_group_size is defined as a constant in src/pipeline/config.py. No magic numbers exist anywhere else in the codebase. To change the ATM window from ±8 strikes to ±10, you edit exactly one line in config.py and the change propagates everywhere.

P3

Atomic Writes — No Partial State on Disk

Both Parquet files and the watermark JSON are written atomically: data is written to a .tmp file first, then renamed to the target path. On macOS APFS, a rename within the same volume is atomic. This means a power failure or crash cannot leave a corrupted half-written file on disk.

P4

IST-Native Time Handling — Zero Conversions

The BigQuery record_time column carries a UTC label, but the numeric values are already IST. Rather than converting and risking a 5:30 shift error, the pipeline strips the UTC label without shifting: pd.to_datetime(df["record_time"]).dt.tz_localize(None). All comparisons against market hours (time(9,15,0), time(15,30,0)) work correctly. Never use dt.tz_convert() on this data.

P5

Separation of Concerns — Three Independent Processes

The pipeline knows nothing about the API. The API knows nothing about the frontend. The frontend knows nothing about Parquet. Each layer talks only to the layer immediately below it, and only through clearly defined interfaces (files on disk, HTTP JSON). Restarting one process never disrupts the others. Upgrading the frontend requires zero changes to the pipeline.
